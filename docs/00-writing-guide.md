# 記載規範

> Status: Active
> 最終更新: 2026-01-23

文章品質、用語、表記に関する規範を定義する。構造・分量・メタ情報・図・命名については[フォーマット規範](./00-format-guide.md)を参照。

---

## 適用範囲

| 対象ファイル | 文章品質 | フォーマット規範 |
|-------------|:--------:|:--------------:|
| `docs/**/*.md` | ✅ | ✅ |
| `README.md` / `CLAUDE.md` | ✅ | - |
| `CHANGELOG.md` / `.github/**` | - | - |

---

## 基本原則

| 原則 | 内容 |
|------|------|
| **コード禁止** | 設計ドキュメントにコードは記載しない |
| **図の積極活用** | Mermaid記法で視覚化。文章だけで説明しない |
| **What/Why優先** | 「何を・なぜ」を書く。「どう実装するか」は書かない |

### コード記載の例外

以下の場合のみ、1-2行程度のコード記載を許可する：

- 設定ファイルの例（`.env`の形式など）
- コマンド実行例（`npm install`など）
- ファイルパスの例示
- 規範ドキュメント内の記法例示（Markdown構文の説明など）

## 文章品質規範

### 1文1情報

1つの文には1つの情報のみ記載する。接続詞で複数の情報を繋げない。

### 主語明示

「〜される」「〜が必要」のような主語不明の文を避ける。動作主体を明記する。

### 肯定形優先

否定形（「〜しない」）より肯定形（「〜する」）を使用する。

### 曖昧表現の制御

| レベル | 表現 | 対応 |
|:------:|------|------|
| 禁止 | 適切に、必要に応じて | 具体的な条件・基準に置き換える |
| 要注意 | 基本的に、原則として | 例外条件を明記するか、この語を削除する |
| 許容 | 例:、主に | 全項目列挙が困難な場合のみ使用 |

#### 要注意表現の使い方

| NG | OK |
|----|-----|
| 基本的にテストを書く | テストを書く。UIプロトタイプは除外する |
| 必要に応じてログを出力する | エラー発生時にログを出力する |

## 自己完結性規範

### 略称定義

略称を初出で使用する際は、正式名称を併記する。例：SSoT（Single Source of Truth）

### 暗黙知の排除

読者がこのドキュメントだけを読んで理解できるように書く。前提知識を仮定しない。

### 参照時の要約併記

他ドキュメントを参照する際は、1文の要約を添える。リンクだけで終わらせない。

```markdown
<!-- OK -->
- [data-model.md](./03-details/data-model.md) - ER図とテーブル定義

<!-- NG -->
- [data-model.md](./03-details/data-model.md)
```

## 冗長性規範

### DRY原則

| 重複の種類 | 判定 | 対応 |
|-----------|:----:|------|
| 概要レベルの要約 | 許容 | 詳細へのリンクを添える |
| 詳細の重複記載 | 禁止 | SSoTを1箇所に定め、他はリンクで参照する |

### フィラー排除

意味のない前置き・つなぎ文を削除する。

| 削除対象 | 理由 |
|---------|------|
| 「以下に〜を示す」 | 見出しで自明 |
| 「前述の通り」 | リンクで参照 |
| 「ここでは〜について説明する」 | 見出しで自明 |

### 装飾語排除

強調副詞は使用しない。事実と数値で表現する。

| NG | OK |
|----|-----|
| 非常に高速に動作する | 応答時間は50ms以下である |
| 極めて重要な設定 | この設定を誤るとデータ消失が発生する |

## 用語・表記規範

### 用語集の使用

- 専門用語・プロジェクト固有語は `99-appendix/glossary.md` で定義する
- 初出時は「〇〇（→用語集）」と明示する

### 表記ゆれ禁止

同じ概念に複数の名前を使わない。用語集で正式名称を定義し、それを使用する。

### 文体

- **である調**で統一（簡潔さ優先）
- 敬語（です・ます調）は使用しない
